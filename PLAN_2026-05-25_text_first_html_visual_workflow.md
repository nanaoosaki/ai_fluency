## Decision

This presentation is intended to be delivered as an HTML5/Reveal.js viewer for full-slide images generated by an image model. The primary source of truth remains the narrative and exact slide text. This repository currently contains the approved story, generation prompts, and an authored HTML preview/fallback; full-slide images will be generated in an environment with image-model access.

The original workflow failed because the image-generation prompts were not grounded in committed on-slide language. The revised workflow keeps the full-slide rendering concept but reverses the order of authorship:

```text
talking points -> exact on-slide text -> full-slide image prompt -> generated slide -> text/visual QA -> static HTML viewer
```

Speaker notes are deferred. They may be added after the visible deck becomes stable, but they are not required to build or rehearse the talk.

## Presentation Thesis

Title:

> Beyond Prompting: Building a Context System That Lets AI Work With You

Claim:

> A model gives capability. A documentation-driven context system makes that capability iterative, reviewable, and reusable.

Audience takeaway:

> Start with one persistent project document. Add what you learn. Organize it when inconsistency begins to slow the work. Then ask AI to act against that maintained context.

## Content Rules

- One argument per screen.
- Slide text is first committed in `STORYBOARD.md`, then supplied verbatim in each full-slide generation prompt.
- Use enough text to carry the concept during free-form speaking, but avoid paragraphs that compete with live delivery.
- Keep high-density demonstration copy concise enough for image generation and inspect every output for text fidelity.
- Use Agent Notes only as the origin of the method, not as a detailed business-system explanation.
- Keep confidential or proprietary data outside the demo.

## Visual Workflow

### Phase 1: Text-Only Story Development

Write the full story and exact slide text before choosing style. The active authored HTML preview is the current review surface until generated slide images are available.

### Phase 2: Prompt Authoring

For every final slide, write a complete generation prompt after the slide claim and visible text are stable. Each prompt must declare:

- the slide claim;
- the visual job;
- composition and negative-space requirements;
- any assets to inject;
- the exact visible text to render in the slide image;
- layout hierarchy and text-fidelity requirements;
- prohibited imagery and visual failure modes.

### Phase 3: Image Generation

Generate complete slide images from approved prompts using an image-enabled OpenAI project or model environment. The current local API project's model catalog was checked on 2026-05-25 and exposes no image-generation model, so rendering is deferred to the company environment expected to provide image access. The prompt files remain provider-legible so the selected image model may be changed without rewriting the story.

### Phase 4: Integration And Review

After images exist, wire accepted generated images into `index.html` as the static presentation. Until then, the active `index.html` remains the authored HTML preview/fallback for narrative review. Inspect each generated slide for exact text, visual hierarchy, readability at 16:9 presentation scale, and continuity across the deck. Regenerate any slide with incorrect or unreadable text.
